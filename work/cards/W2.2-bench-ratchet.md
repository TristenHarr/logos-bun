# W2.2 — P0.8 bench runner + LEDGER.json + the 3σ-separated ratchet (§9.1)

repo: logos-bun. The speed/size/RSS/build-time ratchet. Noise must never deadlock it.

## Deliverable (BAKE_A_BUN §9.1 is the exact arithmetic — follow it precisely)
- `bench/run.mjs`: drives a bench suite via hyperfine (installed), reports `ratio = ours/oracle`
  as a 3-run MEDIAN on pinned hardware, with rolling per-suite noise σ (floor 5%).
- `bench/LEDGER.json`: per-suite locked_ratio + rolling σ window + metric kind. FOUR metric
  kinds, ratcheted identically: wall-clock, peak RSS, binary-size (the shipped `bun`), and
  **build-time** (largo build of logos-bun — the G11 evidence stream, tracked from NOW).
- The two thresholds, separated by >noise so a normal run can't trip the wire:
  - Regression wire = locked_ratio × (1+3σ); a run above it is red ONLY after a confirmatory
    re-run also exceeds it (single-sample outliers never freeze).
  - Win lock: new locked_ratio recorded only when the median improves by >3σ across 3
    consecutive nightlies, and the value locked is the CONSERVATIVE (worse) end. locked_ratio
    only ever decreases.
- LEDGER.json fields join a hash chain (reuse the ledger chain helper if applicable, or its own
  integrity — a hand-edit to loosen a lock must be visible).
- Gate check L12 (bench verify) in gate.sh, house style; empty-suite guard (no locks yet →
  passes trivially). Register the bench-lock guard in gate-manifest.json (coordinate w/ W1.6).

## RED battery
`red/p0/bench/`: synthetic timing data driving the ratchet math —
- a CONFIRMED regression (two samples above the wire) → red;
- a single noise-band blip (one above, re-run below) → NOT red (no false freeze);
- a "win" locks only after 3 consecutive improving runs, and locks the WORSE end (not the
  best-ever sample);
- a hand-edit loosening locked_ratio → detected.
Use injected synthetic measurements (don't actually run hyperfine in the RED test — inject the
3-run medians) so the math is tested deterministically.
